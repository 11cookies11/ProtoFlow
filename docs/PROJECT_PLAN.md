# ProtoFlow 项目计划书（执行版）

## 1. 项目目标

- 将 ProtoFlow 从“可演示”推进到“可稳定交付”版本。
- 打通核心链路：连接设备 -> 收发数据 -> 协议解析 -> 脚本执行 -> 可视化监控。
- 发布一个可安装、可验证、可扩展的 `v1.0`。

## 2. 当前现状（简述）

- 已有：Web UI、Qt WebChannel 桥接、串口/TCP 通信、DSL 执行框架、插件加载框架。
- 待完善：部分功能仍演示化（如部分监控/图表 mock）、模块化能力与测试覆盖不足、端到端稳定性待提升。

## 3. 范围定义

### In Scope

1. 通信核心稳定化（Serial/TCP、异常恢复、日志）
2. DSL 脚本执行闭环（运行、停止、状态反馈）
3. 协议解析能力增强（最少覆盖 Modbus RTU + 可扩展协议）
4. UI 与后端能力对齐（去除假状态、补齐真实数据）
5. 打包发布与文档完善

### Out of Scope（本期不做）

1. 全量 Web 可视化流程编辑器
2. 大规模云端协同功能

## 7. 验收标准（量化）

1. 串口/TCP 连接成功率 > 99%（标准测试环境）。
2. 核心场景连续运行 8 小时无崩溃。
3. DSL 脚本关键流程通过率 100%。
4. UI 核心页面无占位交互、无阻塞级缺陷。
5. 发布包可一键安装并完成基础联调。

## 8. 风险与应对

1. 协议差异导致解析不稳定  
   处理：建立协议样本库 + 回归用例。
2. UI 与后端接口频繁变动  
   处理：冻结桥接接口版本，变更走评审。
3. 设备环境不可控  
   处理：增加模拟器与录包回放测试。

- 例会时间：每周一 10:00（按本地时区）。

## 10. 协作机制（cookies(w) + Codex）

1. 协作主体
   - `cookies(w)`：负责需求决策、优先级、验收结论、发布时间点。
   - `Codex`：负责方案设计、代码实现、联调测试、文档更新、提交落盘。
2. 默认执行流程（适用于后续全部功能）
   - 需求确认 -> 技术实现 -> 本地自测 -> `cookies(w)` 验收 -> 合并发布。
3. 变更规则
   - 无明确暂停指令时，`Codex` 按计划持续推进任务闭环。
   - 关键决策点（范围变化、发布时间、验收标准变更）由 `cookies(w)` 最终确认。
